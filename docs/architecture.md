# Skill Runtime 架构说明

## 这份文档是干什么的

这是一份写给**模板维护者和审计者**的系统说明书。它不负责教普通用户怎么调用某个 Skill，也不描述淘宝、京东等具体业务。

它只回答五个问题：

1. Skill 执行时，谁决定下一步？
2. 仓库里的核心文件分别负责什么？
3. 节点失败、需要确认或需要登录时怎么办？
4. 哪些内容属于稳定核心，不能在运行中自动改变？
5. Skill 如何积累经验，同时不污染稳定核心？

如果只是想创建一个新 Skill，先看根目录的 `README.md`。如果要修改 Runner、工作流协议、学习机制或安全边界，再看本文档。

## 一句话理解整个架构

每个 Skill 都是一条提前定义好的执行路线：

- `workflow.yaml` 是路线图；
- `runner.py` 是唯一调度员；
- Schema 是每一站的验收标准；
- script、MCP、浏览器和推理是不同类型的执行人员；
- `.core-lock.json` 是稳定核心的防漂移封条；
- `learning/` 是与稳定核心隔离的经验笔记本。

Agent 不能凭感觉选择下一步。它只能读取 Runner 返回的当前任务，完成后把结构化结果交回 Runner。

## 为什么需要 Runtime

单纯用一篇 `SKILL.md` 描述流程时，Agent 可能出现以下问题：

- 漏掉中间步骤；
- 在没有确切 SKU 时记录区间最低价；
- 输出格式每次不同；
- 失败后无限重试；
- 没有用户确认就执行外部写操作；
- 把一次偶然经验直接写进长期核心规则。

本架构把这些自由度从 Agent 手中收回，交给确定性的 Runner、Schema、validator 和版本控制。

## 一次完整执行是怎样发生的

```text
用户请求
   ↓
Agent 整理为符合入口 Schema 的 JSON
   ↓
Runner 创建状态，并读取 workflow.yaml 的入口节点
   ↓
Runner 执行 script，或向 Agent 发出一个外部执行指令
   ↓
节点返回结构化 JSON
   ↓
Runner 校验输出 Schema 和 validator
   ↓
通过：进入已声明的下一节点
失败：按配置重试、fallback、等待用户或停止
   ↓
最终输出通过总体验证
   ↓
状态变为 completed
   ↓
记录一条脱敏经验或教训
```

关键点是：**只有 Runner 可以改变当前节点和执行状态。**

## 核心文件各自负责什么

| 文件或目录 | 作用 | 运行时能否自动修改 |
|---|---|---:|
| `SKILL.md` | 告诉 Codex 何时触发 Skill，以及必须通过 Runner 执行 | 否 |
| `workflow.yaml` | 定义节点、顺序、执行器、重试、回退和停止边界 | 否 |
| `schemas/` | 规定每个节点允许接收和返回的数据结构 | 否 |
| `executors/` | 保存固定、重复、可计算的领域脚本 | 否 |
| `validators/` | 对节点结果或最终结果进行确定性检查 | 否 |
| `scripts/runner.py` | 保存状态、执行节点、验证结果并决定下一步 | 否 |
| `scripts/runtime_lib.py` | 提供 Schema、路径、工作流和核心锁验证 | 否 |
| `.core-lock.json` | 记录稳定核心文件的哈希，用于发现未登记变更 | 只能在受审查的核心升级中重建 |
| `learning/ledger.jsonl` | 暂存每次运行产生的一条脱敏经验 | 是 |
| `learning/archive/` | 无损保存已压缩批次的原始学习事件 | 是 |
| `learning/active-rules.json` | 保存数量受限的活跃建议规则 | 是 |
| `learning/proposals/` | 保存将经验晋升为核心的候选提案 | 是，但提案不能直接改核心 |
| `.runtime/` | 保存当前运行状态和节点临时输入输出 | 是，不进入 Git |

领域 Skill 不一定需要预先创建 `executors/` 或 `validators/`。只有真实工作流需要时才添加，避免空目录和无效抽象。

## `workflow.yaml` 到底定义什么

`workflow.yaml` 是 Skill 的执行事实来源。它使用兼容 JSON 的 YAML，因此不需要额外安装 YAML 解析器。

顶层主要包含：

```json
{
  "ir_version": 1,
  "skill_name": "example-skill",
  "configured": true,
  "entry_node": "normalize-input",
  "limits": {
    "max_nodes": 16,
    "total_timeout_seconds": 1800
  },
  "learning": {
    "compact_every": 32,
    "active_rule_limit": 16
  },
  "nodes": [],
  "final_output_schema": "schemas/final.schema.json",
  "final_validator": null
}
```

每个节点必须说明：

- 节点 ID；
- 输入和输出 Schema；
- 使用哪个执行器；
- 精确脚本命令或外部动作；
- 是否会产生副作用；
- 是否必须获得用户确认；
- 超时和最大重试次数；
- 如何验证结果；
- 成功后去哪个节点；
- 失败后是否进入 fallback；
- 什么情况下必须停止。

所有节点都必须能从入口节点到达。成功边和回退边都不能形成循环。重复执行必须使用有上限的重试，不能依赖无限循环。

## 五种执行器怎么选择

设计工作流时按以下顺序选择：

1. `script`：固定、重复、可计算的操作，由 Runner 直接执行。
2. `mcp`：调用具有明确名称和参数结构的外部能力。
3. `browser-dom`：没有 API/MCP 时，通过结构化页面元素操作。
4. `computer-use`：只有无法结构化操作时使用；登录、验证码和扫码交给用户。
5. `reasoning`：只处理无法脚本化的语义判断，但仍必须返回符合 Schema 的 JSON。

这个顺序不是要求每个节点依次尝试五种方式，而是要求设计者优先选择更稳定、可验证的执行器。节点一旦确定执行器，Agent 不能运行时自行替换。

## Runner 如何处理不同情况

Runner 的主要状态如下：

| 状态 | 含义 | 接下来允许做什么 |
|---|---|---|
| `running` | 可以执行当前节点 | 调用 `advance` |
| `waiting-confirmation` | 当前节点有写入或破坏性副作用 | 等待真实用户确认，再调用 `approve` |
| `waiting-external` | 等待 MCP、浏览器、Computer Use 或 reasoning 返回结果 | 执行已声明动作，再调用 `submit` |
| `waiting-user` | 需要用户登录、扫码或补充信息 | 用户完成后调用 `resume` |
| `completed` | 最终结果已通过验证 | 可以交付并记录一条经验 |
| `failed` | 出现致命错误或重试耗尽 | 停止，不得伪装成完成 |

脚本节点由 Runner 自己执行，使用 argv 数组和 `shell=false`。外部节点由 Runner 发出结构化指令，Agent 只能执行指令中声明的动作，并提交符合输出 Schema 的 JSON。

## 失败、回退和停止的区别

- **重试**：同一节点发生临时错误，可以在 `max_retries` 范围内再次执行。
- **fallback**：节点无法成功时，进入工作流预先声明的替代节点。
- **等待用户**：流程暂时无法继续，但用户完成登录或补充信息后可以恢复。
- **停止**：触发安全边界、致命错误或重试耗尽，状态变为 `failed`。

文字形式的停止条件只能约束外部执行器。能够由机器判断的停止条件，必须写进 Schema 或 validator，不能假装 Runner 能理解任意自然语言规则。

## 稳定核心与成长系统如何隔离

稳定核心包括：

- 工作流；
- Schema；
- 脚本和 validator；
- Skill 指令；
- 权限、确认和安全边界；
- Runtime 与强制验证配置。

运行时学习不能修改这些内容。每次完成或用户暂停的运行，只能向 `learning/` 写入一条脱敏经验。

默认累计 32 条事件后：

1. 原始事件完整写入归档；
2. 重复经验进行确定性合并；
3. 活跃建议规则限制为最多 16 条；
4. 未进入活跃上下文的原始信息仍保存在归档和 Git 历史中。

活跃规则只能提供建议，不能改变节点顺序、权限、确认要求、validator 或停止条件。经验要进入稳定核心，必须先生成提案，再经过反例审查、回归测试、版本升级、人工批准和重新冻结。

## 核心锁能做什么、不能做什么

`.core-lock.json` 会记录稳定核心文件的 SHA-256 哈希。Runner 在开始或恢复任务前验证这些哈希；不一致时直接停止。

它能够发现：

- 核心文件被意外编辑；
- 工作流已经改变但没有重新冻结；
- 脚本、Schema 或安全规则与已确认版本不一致。

它不能证明修改者身份，也不是防黑客机制。拥有本地写权限的进程仍可以修改文件并重新生成锁。因此还需要 Git 历史、受保护分支、代码审查、固定依赖的 CI，以及风险较高时的签名发布。

Runner 的确认状态也具有同样边界：它能阻止流程自动继续，但宿主和 Agent 仍必须确保只有在用户真实确认后才调用 `approve`。

## 为什么新仓库默认不能运行

新生成仓库的 `workflow.yaml` 设置为：

```json
"configured": false
```

这表示仓库只有通用 Runtime，还没有真正的领域工作流。Runner 会拒绝运行，防止一个空泛的 reasoning 节点冒充已经完成的 Skill。

只有完成以下工作后才能改为 `true`：

1. 明确 Skill 的最小支持范围和排除范围；
2. 设计实际节点与执行器；
3. 完成每个节点的 Schema；
4. 添加成功、失败和相邻非触发测试；
5. 通过安全审查并重新生成核心锁。

## 审计这套架构时看什么

审计者应逐项确认：

- 文档中的每条强制规则是否真的有代码实现；
- Agent 是否能跳过 Runner、节点或 validator；
- 写入操作是否确实停在确认状态；
- 重试、fallback 和总执行时间是否都有上限；
- 学习脚本是否只能写入 `learning/`；
- 原始事件是否在截断账本前完成归档；
- 核心变更是否必然导致锁不匹配；
- 每个安全承诺是否至少有一个失败路径测试。

如果一条规则只能依靠 Agent “自觉遵守”，文档必须明确标注它是协议约束，而不能把它描述为不可绕过的技术边界。
