# 版本与维护入门

## 这份文档解决什么问题

这份文档说明一个 Skill 何时需要改版本、修改后要检查什么、怎样保留回滚能力

目标读者是维护已经生成的 Skill 仓库的人

读完以后，你应该能够选择版本号、完成发布检查，并为故障保留明确回滚点

如果你只想第一次创建 Skill，请先阅读根目录的 `README.md`

## 先理解四个词

### 提交

提交是一次带说明的 Git 保存点

它让维护者知道哪些文件在同一次修改中发生了变化

### 版本

版本是写在 `VERSION` 文件中的公开编号

本模板使用 `主版本.次版本.修订版本`，例如 `1.4.2`

### 标签

标签是 Git 中指向某次提交的固定名称，例如 `v1.4.2`

标签让维护者可以准确找到一个已经发布的版本

### Release

Release 是托管平台上的发布页面

它通常关联一个标签，并向使用者解释这个版本新增了什么、修复了什么、是否需要迁移

## 每个 Skill 都独立维护

模板工厂只负责生成起点

每个生成后的 Skill 都拥有独立的仓库、版本、测试、学习记录、提交、标签、Release 和回滚历史

不要把多个生成后的 Skill 仓库放回本模板仓库

## 什么是公开契约

公开契约是使用者能够依赖的稳定行为

它至少包括以下内容

- 支持哪些用户意图
- 明确不支持哪些相邻意图
- 需要哪些输入
- 返回哪些输出
- 允许调用哪些执行器
- 可能产生哪些副作用
- 哪些操作必须等待用户确认
- 失败时何时重试、回退或停止
- Runner、Schema 和 validator 提供哪些强制保证

只要这些内容发生变化，就需要判断版本影响

## 如何选择版本号

| 变化类型 | 使用的版本 | 判断方法 | 例子 |
| --- | --- | --- | --- |
| 修订版本 | PATCH | 旧用法保持有效，公开契约没有变化 | 修复脚本错误、补充说明、增强不改变结果的校验 |
| 次版本 | MINOR | 旧用法保持有效，同时增加可选能力 | 增加一个可选节点、增加一个可选输入、支持新的非破坏性输出字段 |
| 主版本 | MAJOR | 旧用法可能失效，或者权限与风险边界发生变化 | 修改必需输入、改变输出结构、增加写入副作用、改变节点顺序 |

例如当前版本是 `1.4.2`

- PATCH 后是 `1.4.3`
- MINOR 后是 `1.5.0`
- MAJOR 后是 `2.0.0`

只修改学习数据不会自动改变核心版本

把学习规则晋升为稳定核心时，必须根据公开契约的变化选择新版本

## Skill 的四个阶段

### 草稿 Draft

`workflow.yaml` 中的 `definition.configured` 为 `false`

此阶段可以检查文件结构和 Schema，Runner 会拒绝真实执行

### 候选 Candidate

工作流已经配置完成，并且具备成功路径测试、失败路径测试和相邻非触发测试

此阶段用于受控试运行

### 稳定 Stable

Skill 已经通过重复任务评估、真实低风险使用、完整验证和核心锁检查

此阶段应当拥有版本标签和明确回滚点

### 已弃用 Deprecated

Skill 已经不再推荐使用

移除前需要关闭隐式触发，说明替代方案，并给使用者留出迁移时间

## 修改前要做什么

1. 阅读 `SKILL.md`，确认触发范围
2. 阅读 `workflow.yaml`，确认修改会影响哪些节点
3. 阅读 `SECURITY.md`，确认权限和敏感信息边界
4. 检查当前工作区，避免覆盖尚未提交的修改
5. 写清楚本次修改要改变的公开行为
6. 预先判断版本属于 PATCH、MINOR 还是 MAJOR

## 修改后要做什么

### 第一步：运行允许核心变化的验证

```bash
python3 scripts/validate.py --ignore-core-lock
```

这个命令检查仓库结构、工作流、Schema、脚本和配置

`--ignore-core-lock` 只在维护核心时使用，因为旧核心锁还没有包含新内容

### 第二步：运行测试

```bash
python3 -m unittest discover -s tests -v
```

测试应当覆盖成功路径、预期失败、确认关卡、停止条件和本次修复对应的回归案例

### 第三步：扫描敏感信息

```bash
python3 scripts/check_secrets.py
```

发现凭据、会话信息或个人数据时立即停止提交

### 第四步：更新版本和变更记录

修改 `VERSION`

在 `CHANGELOG.md` 中说明新增、修改、修复、弃用或安全变化

变更记录应当面向使用者解释影响，不要只罗列文件名

### 第五步：重新生成核心锁

```bash
python3 .agents/skills/<skill-name>/scripts/freeze_core.py
```

核心锁保存稳定文件的摘要

它用于发现未经说明的漂移，不用于证明修改者身份

### 第六步：运行最终验证

```bash
python3 scripts/validate.py
python3 .agents/skills/<skill-name>/scripts/freeze_core.py --check
```

最终验证必须在不忽略核心锁的情况下通过

## 发布前检查

- `VERSION` 与计划发布的标签一致
- `CHANGELOG.md` 已解释用户可见变化
- 成功路径和失败路径测试全部通过
- 相邻但不应触发的请求仍然不会触发
- 新增副作用已经声明，并设置所需确认
- 敏感信息扫描没有发现问题
- `.core-lock.json` 与当前稳定核心一致
- 回滚目标是一个已知可用的提交或标签

## 出现问题时怎样回滚

先停止继续发布和扩大使用范围

找到上一个已知可用的标签

保留故障现场和必要的脱敏证据

使用新的修复提交恢复行为，不要静默改写已经发布的事实

如果问题涉及泄露的凭据，先撤销或轮换凭据，再处理 Git 历史

## 什么时候必须停止维护操作

- 公开契约变化无法确定
- 修改需要新的外部权限，但没有获得授权
- 测试无法证明关键行为
- 核心锁差异无法解释
- 扫描发现敏感信息
- 回滚目标不存在或未验证

发生这些情况时，先解决不确定性，再提交或发布

## 最后自检

维护者应当能够回答以下问题

1. 这次修改改变了什么行为
2. 为什么选择当前版本号
3. 哪些测试证明修改有效
4. 哪些文件属于稳定核心
5. 失败后回到哪个版本
6. 是否新增权限、副作用或用户确认
