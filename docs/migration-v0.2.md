# 从 v0.1 迁移到当前架构

## 这份文档解决什么问题

这份文档帮助旧版 v0.1 Skill 的维护者，把一个旧 Skill 迁移成当前的独立仓库结构；

适合阅读这份文档的人是仍在维护 v0.1 Skill 的仓库所有者和迁移执行者；

读完以后，你应该能够逐项搬迁触发范围、节点、安全规则和测试，并完成受控切换；

新创建的 Skill 不需要执行迁移，请直接阅读根目录的 `README.md`；

## 先理解迁移目标

v0.1 把多个 Skill 放在同一个本地仓库，并使用共享 catalog、四种 profile 和共享评估目录；

标签 `v0.1.0` 保留了旧设计，便于查看和回退；

当前架构为每个 Skill 创建独立 Git 仓库；

每个仓库拥有自己的工作流、测试、安全边界、学习记录、版本和发布历史；

## 新旧架构对照

| v0.1 | 当前架构 | 迁移后的作用 |
| --- | --- | --- |
| 一个仓库存放多个 Skill | 一个仓库只存放一个 Skill | 修改、发布和回滚互不影响 |
| profile 决定模板变体 | 所有 Skill 使用同一基础模板 | 差异写入 Skill 自己的工作流 |
| catalog 保存集中元数据 | 不使用内部 catalog | 仓库自身就是管理边界 |
| Agent 自行决定执行顺序 | Runner 按 `workflow.yaml` 推进 | 执行顺序可以验证和复现 |
| 说明文字承担安全约束 | Schema、validator、确认和停止条件强制约束 | 关键边界由机器执行 |
| 学习内容可能进入共享区域 | 每个 Skill 只维护自己的 `learning/` | 学习记录不会跨 Skill 污染 |
| 学习可能直接影响核心 | 先记录建议，再人工晋升 | 成长过程可以审计和回滚 |

## 迁移前准备

1. 确认旧 Skill 的仓库和版本；
2. 保存一个只读备份或创建迁移前标签；
3. 收集已知成功案例、失败案例和不应触发的相邻请求；
4. 列出旧 Skill 使用的脚本、工具、权限和副作用；
5. 删除资料中的凭据、Cookie、会话、地址和其他敏感信息；

旧仓库在迁移完成前保持可用；

## 迁移步骤

### 第一步：创建新的独立仓库

使用模板工厂生成一个新仓库；

Skill 名称应当保持稳定，路由描述要明确写出支持意图和相邻非意图；

新仓库初始为安全草稿，`definition.configured` 保持 `false`；

### 第二步：迁移触发范围

把旧 Skill 的用途写入新仓库的 `SKILL.md`；

只保留一个清晰任务边界；

对容易混淆的相邻请求明确写出不触发条件；

### 第三步：把旧流程转换为节点

打开 `workflow.yaml`；

把旧流程中的每个必要步骤转换为一个节点；

每个节点都要声明输入、输出、执行器、副作用、确认要求、停止条件、重试和失败去向；

执行顺序只写在工作流图中，不依赖 Agent 临时决定；

### 第四步：迁移固定操作

重复、机械、可计算的操作迁移为 `script` 节点；

脚本使用结构化 JSON 输入和输出；

脚本命令使用 argv 数组，并保持 `shell=false`；

每个脚本都需要对应的输入 Schema、输出 Schema 和测试；

### 第五步：迁移外部操作

结构化服务调用使用 `mcp`；

网页结构可可靠读取时使用 `browser-dom`；

只有缺少结构化交互时才使用 `computer-use`；

无法机械判断的语义任务使用 `reasoning`；

外部节点必须把结果作为符合输出 Schema 的 JSON 提交给 Runner；

### 第六步：迁移安全规则

把旧版的每条安全说明转换为可执行字段；

- 会改变外部状态的行为写入副作用；
- 需要用户同意的行为写入确认关卡；
- 不允许继续的情况写入停止条件；
- 可以自动检查的结果写入 validator；
- 可以安全替代的路径写入 fallback；

文字说明可以帮助 Agent 理解，关键边界必须由 Runner、Schema 或 validator 强制执行；

### 第七步：迁移参考资料

只迁移当前 Skill 运行时真正需要的资料；

不要复制旧 catalog、共享评估控制面、无关案例或历史敏感数据；

参考资料只向需要它的节点开放；

### 第八步：迁移测试

至少加入以下测试；

- 一个正常完成的案例；
- 一个输入不合法的案例；
- 一个节点输出不合法的案例；
- 一个需要确认的案例；
- 一个触发停止条件的案例；
- 一个相邻但不应触发的案例；
- 每个旧故障对应的回归案例；

### 第九步：完成配置

所有节点、Schema、脚本和测试完成后，把 `definition.configured` 改为 `true`；

更新 `VERSION` 和 `CHANGELOG.md`；

运行验证、测试和敏感信息扫描；

重新生成 `.core-lock.json`；

### 第十步：受控切换

先使用低风险真实任务验证新 Skill；

比较旧版与新版的结果、失败方式和用户确认点；

确认新版本稳定后创建版本标签；

保留旧版回滚点，直到迁移观察期结束；

## 不要迁移的内容

- 旧 catalog；
- 共享 profile；
- 共享评估控制面；
- 未经验证的学习结论；
- 原始用户提示；
- 完整工具输出；
- 凭据、Cookie、会话和个人信息；
- 与当前 Skill 无关的参考资料；

## 迁移完成检查

- 新仓库只包含一个 Skill；
- `workflow.yaml` 是执行流程的唯一事实来源；
- Runner 能够完成成功路径；
- Runner 能够拒绝非法输入和非法输出；
- 所有外部写入都要求适当确认；
- 所有停止条件都能让状态安全暂停；
- 测试覆盖旧版成功和失败案例；
- `definition.configured=true`；
- 完整验证和核心锁检查通过；
- 新旧版本都具有明确标签或提交；
- 没有复制 catalog 或敏感信息；
