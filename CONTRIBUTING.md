# 修改模板前先读这里

## 这份文件能帮你做什么

这份文件回答一个问题：怎样安全地修改通用 Skill 模板？

适合阅读这份文件的人：

- 准备修改模板代码的人；
- 准备修改模板文档的人；
- 准备审查一次模板变更的人；

读完以后，你应该能够完成一次小范围修改，并判断它是否可以提交；

如果你只想使用模板创建 Skill，可以暂时跳过这份文件，先阅读根目录的 `README.md`；

## 为什么修改模板需要谨慎

这个仓库会生成新的 Skill 仓库；

模板中的错误会被复制到以后创建的 Skill 中；

运行流程、安全规则和生成脚本发生变化时，影响范围会更大；

因此，每次修改都要保持范围清楚、结果可验证、失败可以定位；

## 一次修改包含哪些步骤

一次完整修改分为七步：

1. 写清楚准备改变什么；
2. 判断它会影响哪些公开行为；
3. 只修改完成目标所需的文件；
4. 为行为变化补充测试；
5. 运行全部检查；
6. 阅读最终差异；
7. 提交并推送经过验证的内容；

下面逐步说明；

## 第一步：写清楚准备改变什么

开始编辑前，先写出一张变更说明卡；

```text
目标：这次要解决什么问题？
范围：准备修改哪些文件？
保持不变：哪些行为不能受到影响？
成功：看到什么结果才算完成？
失败：最接近的错误情况是什么？
```

这张卡用于限制修改范围；一个目标无法用几句话说明时，应当继续拆分任务；

## 第二步：判断公开行为是否变化

公开行为是 Skill 使用者能够观察或依赖的内容；

常见公开行为包括：

- Skill 会在什么请求下触发；
- 用户必须提供哪些输入；
- 最终输出包含哪些内容；
- 工作流按照什么顺序执行；
- 节点可以使用哪些执行器；
- 哪些操作需要用户确认；
- 失败后会重试、回退还是停止；

公开行为发生变化时，需要判断版本影响；

| 版本类型 | 什么时候使用 | 简单例子 |
|---|---|---|
| PATCH | 修复实现或说明，现有使用方式保持有效； | 修正文档解释或修复不会改变输出结构的错误； |
| MINOR | 增加兼容能力，现有使用方式仍然有效； | 增加一个可选节点或兼容的结果检查； |
| MAJOR | 现有使用方式需要迁移； | 改变必填输入、输出结构、权限或节点顺序； |

需要升级版本时，同时更新 `VERSION` 和 `CHANGELOG.md`；

纯文档澄清没有改变运行行为时，可以保持版本不变；提交说明中要写清理由；

## 第三步：保持修改范围最小

只编辑变更说明卡中列出的文件；

发现新的独立问题时，先记录它，再放到下一次修改中处理；

这样可以让审查者准确判断每一行变化的原因；

工作区中存在其他人的修改时，保留这些内容；无法分离修改范围时，停止并向仓库所有者确认；

生成后的 Skill 仓库必须放在本模板仓库之外；模板仓库内部不能出现嵌套的生成仓库；

## 第四步：为行为变化补充测试

运行行为发生变化时，至少增加两个测试：

1. 成功测试：证明新的预期行为可以完成；
2. 邻近失败测试：证明最接近的错误输入或错误状态会被拒绝；

例如，新增一个必填工作流字段时：

- 成功测试使用合法字段值，并确认验证通过；
- 失败测试删除该字段，并确认验证器返回明确错误；

测试应当检查外部可观察结果；测试不能只确认某一行内部代码被调用；

## 第五步：运行全部检查

在仓库根目录依次运行下面三条命令；

仓库根目录是包含根目录 `README.md` 的最外层文件夹；

### 检查模板能否生成仓库

```bash
python3 scripts/validate_template.py
```

这条命令会检查必需文档、中文正文标点、Python 语法、占位符和临时生成流程；

占位符是生成仓库时才会替换的临时文字，例如 `__SKILL_NAME__`；

成功时会看到：

```text
Universal one-Skill repository template is valid.
```

失败时，先处理输出中列出的文件或占位符问题，再重新运行；

### 运行 Runtime 和生成器测试

```bash
python3 -m unittest discover -s tests -v
```

这条命令会执行成功路径和失败路径测试；

成功时，最后会看到 `OK`；任何 `FAILED` 或 `ERROR` 都表示当前修改不能提交；

### 扫描敏感信息

```bash
python3 scripts/check_secrets.py
```

这条命令会检查凭据、令牌、私钥和其他不应进入 Git 的内容；

成功时会看到敏感信息扫描通过；

发现命中项时，先移除真实敏感值；已经暴露的凭据需要立即撤销或轮换；

## 第六步：阅读最终差异

提交前运行：

```bash
git status -sb
git diff
```

`git status -sb` 显示哪些文件发生了变化；

`git diff` 显示每一行具体改了什么；以 `-` 开头的行是旧内容，以 `+` 开头的行是新内容；

你需要确认：

- 每一行变化都服务于本次目标；
- 没有残留调试内容；
- 没有意外修改其他文件；
- 文档描述与实际代码一致；
- 新增示例没有真实凭据或个人信息；

发现无法解释的变化时，不要提交；先确认它的来源和作用；

## 第七步：准备证据并提交

每次修改需要留下足够的审查证据；

| 证据 | 需要回答的问题 |
|---|---|
| 运行规则 | 哪条 Runtime 规则发生了变化？ |
| 成功测试 | 什么测试证明目标已经实现？ |
| 失败测试 | 什么测试证明错误情况会被拒绝？ |
| 草稿安全 | 新生成的草稿仓库是否仍然不可执行？ |
| 核心锁 | 未冻结的稳定核心修改是否仍会触发硬停止？ |
| 学习边界 | 学习功能是否仍然只能写入 `learning/`，并完整保留归档事件？ |
| 版本影响 | 这是 PATCH、MINOR、MAJOR，还是无需升级版本的文档澄清？ |
| 文档解释 | 新术语和新字段是否足以让目标读者独立理解？ |

某项证据与本次修改无关时，写明“不适用”并说明原因；全部仓库检查仍然需要运行；

证据准备完成后，只暂存本次修改的文件；

```bash
git add "需要提交的文件路径"
git commit -m "简短说明"
git push origin "当前分支名"
```

引号中的中文说明是占位符，执行前需要换成真实内容；

例如，本次只修改 `CONTRIBUTING.md` 时，第一条命令应写成 `git add CONTRIBUTING.md`；

当前模板仓库按照仓库所有者的要求直接推送 `main`，因此第三条命令使用 `git push origin main`，不创建 PR；

不要使用会顺便加入全部文件的命令；这样可以避免把无关修改提交到同一个版本中；

提交成功时，Git 会显示新的提交编号；推送成功时，远端仓库会出现同一个提交；

提交失败时，先阅读错误信息；

常见原因包括没有暂存文件、身份验证失效、远端已有更新或分支名称错误；

## 文档必须怎样写

人类阅读的文档默认读者没有相关技术背景；

### 每份文档先说明阅读目的

文档开头必须回答：

- 这份文件解决什么问题？
- 谁需要阅读？
- 读完以后会做什么？
- 哪些人可以暂时跳过？

### 一个小节只解释一个主要概念

先解释当前概念，再引入下一个概念；

普通段落保持一到三句话；长流程拆成编号步骤，精确映射使用表格；

### 新术语需要五项信息

技术术语第一次出现时，所在小节必须依次说明：

1. 它是什么；
2. 它解决什么问题；
3. 它长什么样；
4. 谁负责读取、修改或执行；
5. 失败或越界时会发生什么；

这五项可以分成多个短段落；不要压缩成一个长句；

### 命令需要完整教学信息

每个命令都要说明：

1. 为什么运行；
2. 谁来运行；
3. 运行前需要什么；
4. 完整命令是什么；
5. 成功输出长什么样；
6. 失败后检查什么；

命令中的占位符必须在代码块附近解释；

### 使用清楚、确定的表达

人类文档使用简体中文；字段名、命令和文件名保留真实英文标识符；

优先使用短句和肯定句；避免先否定再转折的表达；通用架构文档使用领域无关的示例；

由 Agent 判断的文字协议和由程序执行的机器规则必须分开说明；

### Agent 指令保持简洁

`AGENTS.md` 和 `SKILL.md` 主要由 Agent 读取；

这两类文件只保留执行所需的明确规则；面向人类的背景解释放入 README 或 `docs/`，并提供清楚链接；

## 人工审计文档时问什么

审计者只需要回答五个问题：

1. 这份文件是干什么的？
2. 我现在应该做什么？
3. 当前操作由我、Agent 还是 Runner 完成？
4. 成功后我会看到什么？
5. 失败后我应该怎么办？

任何一个问题无法只靠当前页面回答时，文档需要继续修改；

## 什么情况下必须停止

遇到以下情况时，停止修改并请求仓库所有者决定：

- 目标存在多种解释，并且不同解释会改变公开行为；
- 修改需要扩大到变更说明卡之外的多个独立问题；
- 工作区包含无法安全分离的其他修改；
- 测试失败原因无法确定；
- 可能已经提交或输出真实敏感信息；
- 无法用简单语言解释新增规则的用途和失败结果；

## 完成标准

满足以下条件后，本次修改才算完成：

- 目标已经实现；
- 成功和失败路径已经验证；
- 三项仓库检查全部通过；
- 最终差异中没有无关内容；
- 版本影响已经说明；
- 文档通过五个问题的人工审计；
- 提交和推送范围与本次目标一致；
