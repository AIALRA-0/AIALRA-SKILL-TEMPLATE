# 安全策略

## 这份文件能帮你做什么

这份文件回答三个问题：哪些内容不能提交、模板提供哪些保护、发生泄漏后先做什么；

准备修改模板、生成仓库或推送 GitHub 的人都需要阅读；

读完以后，你应该能够识别敏感信息，并完成提交前安全检查；

## 哪些内容不能进入 Git

以下内容禁止提交：

- API 密钥；
- 访问令牌；
- 密码；
- Cookie；
- 浏览器配置和会话导出；
- 私钥；
- 地址、订单历史和个人身份信息；
- 私人消息；
- 未脱敏的运行日志；
- 真实系统返回的完整敏感数据；

示例需要敏感值时，使用 `${SERVICE_TOKEN}` 这类明确占位符；

示例需要域名时，使用 `.invalid` 域名；

## 登录信息应该保存在哪里

身份验证保留在用户已经授权的工具或浏览器会话中；

登录、验证码、二维码和双重认证由用户亲自完成；

Skill 仓库只保存调用协议和字段结构；它不保存可复用的真实登录状态；

## 模板提供哪些保护

### 输入和输出检查

Runner 使用 JSON Schema 检查初始输入、每个节点输出和最终输出；

结构不符合 Schema 时，Runner 拒绝结果并进入预设失败流程；

### 脚本命令边界

脚本节点使用 argv 数组和 `shell=false`；

Runner 直接启动指定程序，并保持每个参数的边界；Shell 不会自动解释管道、重定向和命令替换字符；

### 外部动作边界

外部节点必须声明执行器、动作、参数、输出 Schema、超时和停止条件；

Agent 或宿主只能执行 Runner 返回的当前动作；

### 用户确认

`write` 和 `destructive` 节点执行前必须进入确认状态；

宿主只有在用户真实同意后才能调用 `approve`；

### 资源限制

节点数量、总运行时间、单节点超时、重试次数和回退路径都受到限制；

预算耗尽时，工作流进入失败状态；

### 核心锁

`.core-lock.json` 保存稳定核心文件的 SHA-256 清单；

文件内容与清单不一致时，Runner 硬停止；

核心锁用于检测漂移；代码来源由 Git 历史、仓库权限、分支保护和签名发布共同保证；

### 学习边界

学习脚本只能写入 `learning/`；

学习事件必须脱敏，并使用受控证据前缀；学习结果不能自动修改稳定核心；

## 推送前运行什么

在仓库根目录运行：

```bash
python3 scripts/check_secrets.py
```

这条命令检查当前工作区中的常见凭据和私钥模式；

成功时会看到敏感信息扫描通过；

CI 还会下载固定版本和校验和的 Gitleaks，并扫描完整 Git 历史；

GitHub 仓库应当启用远端 Secret Scanning 和 Push Protection；

## 发现敏感信息后先做什么

按照以下顺序处理：

1. 立即撤销或轮换已经暴露的凭据；
2. 停止继续使用旧凭据；
3. 删除当前文件中的敏感值；
4. 检查 Git 历史、CI 日志和其他副本；
5. 根据仓库所有者决定清理历史；
6. 重新运行本地扫描和远端扫描；

删除当前文件只能移除最新副本；已经提交或输出的凭据仍然需要撤销或轮换；

## 什么时候必须停止

遇到以下情况时，停止推送并请求仓库所有者处理：

- 无法判断某段内容是否包含真实凭据；
- 扫描器持续报告命中；
- 敏感信息可能已经进入提交历史；
- 外部写入没有取得用户确认；
- 核心锁出现无法解释的差异；
- 安全检查需要被关闭才能通过；
